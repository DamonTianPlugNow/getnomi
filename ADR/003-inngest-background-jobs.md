# ADR-003: 使用 Inngest 处理后台任务

## 状态

Accepted

## 日期

2024-03-13

## 背景

项目需要处理多种后台任务：

- AI Profile 生成（耗时操作）
- 匹配算法执行
- 定时任务（每日匹配、过期检查）
- 会议创建和提醒
- 邮件发送

这些任务需要：

- 异步执行，不阻塞用户请求
- 失败重试
- 任务调度
- 可观测性

## 决策

使用 Inngest 作为后台任务框架。

## 选项考虑

### 选项 1: Inngest

- 优点：
  - 事件驱动，声明式 API
  - 内置重试、调度、并发控制
  - 优秀的开发体验（本地 Dev Server）
  - 与 Vercel 无缝集成
  - 可视化任务监控
- 缺点：
  - 相对较新
  - 免费额度有限

### 选项 2: Vercel Cron + Edge Functions

- 优点：原生集成，无需额外服务
- 缺点：
  - 只支持定时任务
  - 无重试机制
  - 执行时间限制

### 选项 3: BullMQ + Redis

- 优点：成熟稳定，功能强大
- 缺点：
  - 需要自己管理 Redis
  - Serverless 环境不友好
  - 需要额外的 Worker 进程

### 选项 4: AWS SQS + Lambda

- 优点：高可用，可扩展
- 缺点：
  - 配置复杂
  - 与 Vercel 集成不便
  - 成本不透明

### 选项 5: Trigger.dev

- 优点：类似 Inngest，开源
- 缺点：生态较小，文档不如 Inngest 完善

## 理由

1. 事件驱动模式非常适合我们的业务流程
2. 开发体验优秀，本地调试方便
3. 与 Vercel 部署无缝集成
4. 内置的重试和调度满足所有需求
5. 可视化监控便于排查问题

## 影响

- 正面：开发效率高，可靠性好
- 负面：增加了一个外部依赖
- 风险：免费额度用完后需要付费
