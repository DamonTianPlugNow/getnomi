# ADR-004: 多 Agent 架构设计

## 状态

Accepted

## 日期

2024-03-14

## 背景

项目的核心功能是通过 AI 对话了解用户，生成用户画像。需要设计 AI 系统架构来：

- 自然地引导对话
- 从对话中提取有价值的信息
- 生成多维度的用户画像
- 支持匹配算法

## 决策

采用多 Agent 架构，包含：

- Orchestrator（编排器）
- Classifier（分类器）
- 专业 Agents（Emotional、Persona、UserStyle）
- Memory 系统

## 选项考虑

### 选项 1: 单一 Prompt 方案

- 优点：简单，延迟低
- 缺点：
  - Prompt 过长，难以维护
  - 难以处理复杂的多轮对话
  - 输出质量不稳定

### 选项 2: 多 Agent 架构

- 优点：
  - 职责分离，易于维护
  - 每个 Agent 专注特定任务
  - 可以独立优化各 Agent
  - 输出更稳定可控
- 缺点：
  - 架构复杂
  - 多次 API 调用，成本和延迟增加

### 选项 3: LangChain / LlamaIndex

- 优点：成熟的框架，功能丰富
- 缺点：
  - 过度抽象，学习成本高
  - 依赖重，bundle size 大
  - 灵活性受限

### 选项 4: 使用 Claude 的 Tool Use

- 优点：原生支持，可靠性高
- 缺点：
  - 不适合我们的场景（分析而非执行）
  - 控制流程不够灵活

## 理由

1. 多 Agent 架构符合"单一职责"原则
2. 专业 Agent 可以针对性优化 Prompt
3. Memory 系统支持长对话上下文
4. 架构可扩展，未来可以添���更多 Agent
5. 不依赖重型框架，保持代码可控

## 架构设计

```
用户输入
    │
    ▼
┌─────────────┐
│ Classifier  │ ─── 意图识别、情感检测
└─────────────┘
    │
    ▼
┌─────────────┐
│ Orchestrator│ ─── 协调各 Agent，生成响应
└─────────────┘
    │
    ├──▶ EmotionalAgent ─── 情感模式分析
    ├──▶ PersonaAgent ───── 人格特征分析
    └──▶ UserStyleAgent ─── 沟通风格分析
    │
    ▼
┌─────────────┐
│   Memory    │ ─── 存储关键信息
└─────────────┘
```

## 影响

- 正面：
  - 代码结构清晰
  - 各模块可独立测试
  - 输出质量可控
- 负面：
  - 开发复杂度增加
  - API 调用成本增加
- 风险：
  - 多 Agent 协调可能出现不一致
